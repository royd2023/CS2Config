- Background: solid dark (#1a1a1a or similar) — mimics CS2's dark environment

- Style picker is a segmented button group (4 buttons in a row) at the top of the controls section

- Color input: separate R, G, B sliders + separate alpha slider — matches CS2's internal value format
- A small color swatch sits beside the sliders, showing the current color with checkerboard behind it to indicate alpha

## Summary

Phase 2 assembles the crosshair designer UI from parts that are nearly all defined or already built. The Zustand store (CrosshairState), the share code encoder (crosshairCodeGen), and the four UI primitives (Slider, Select, ColorPicker, CopyButton) are complete from Phase 1. Phase 2 adds five new components: CrosshairPreview (SVG renderer), SegmentedButtonGroup (style picker), ColorSection (RGBA sliders + swatch), ToggleSwitch (boolean params), and ImportCodeBar (always-visible code + copy). These compose into a single CrosshairDesigner page component wired into App.tsx.

The most technically involved piece is the SVG rendering math. The UI-SPEC defines a concrete scale rule (1 CS2 unit = 4px), which eliminates the empirical-validation concern logged in STATE.md blockers. The library source has been fully inspected — there are no surprises in the akiver/csgo-sharecode API beyond a CS2-specific style numbering difference noted below.

**Primary recommendation:** Build each new component in isolation (pure props, no store access), then compose them in CrosshairDesigner which owns all store reads. This keeps every leaf component easily testable with React Testing Library.

## SVG Rendering Math

[VERIFIED: UI-SPEC.md CrosshairPreview section — scale rule is locked]

### Scale rule

```
1 CS2 unit = 4px at 1080p reference
cl_crosshairsize 4 → 16px arm length from center
cl_crosshairgap → maps 1:1 to px (negative values pull arms inward)
cl_crosshairthickness → maps 1:1 to px (minimum 1px rendered)
```

### SVG coordinate model

The SVG viewBox should be symmetric around a center point. A safe canvas for the default crosshair (size 4, gap -3, thickness 1) at 4× scale:

```
armLength = line.length * 4          // e.g. 4 * 4 = 16px
gap = line.gap                       // e.g. -3px (negative = arms overlap center)
thickness = Math.max(1, line.thickness)

// arm positions from center (0,0):
// right arm: x from (gap) to (gap + armLength)
// left arm:  x from -(gap + armLength) to -gap
// top arm:   y from -(gap + armLength) to -gap   [omitted when tStyle=true]
// bottom arm: y from gap to (gap + armLength)
```

Recommended SVG canvas: 200×200px with `viewBox="-100 -100 200 200"` so center is always at (0,0). This accommodates all practical parameter combinations.

### Arm rectangles (4 lines for standard crosshair)

Each arm is an SVG `<rect>`. Using center-origin coordinate system:

```typescript
// Right arm
{ x: gap, y: -thickness/2, width: armLength, height: thickness }

// Left arm
{ x: -(gap + armLength), y: -thickness/2, width: armLength, height: thickness }

// Bottom arm
{ x: -thickness/2, y: gap, width: thickness, height: armLength }

// Top arm (omit when tStyle === true)
{ x: -thickness/2, y: -(gap + armLength), width: thickness, height: armLength }
```

### Center dot

Rendered as a square `<rect>` at center, size derived from thickness:

```typescript
// Square dot at center, same thickness as lines
const dotSize = thickness * 2  // reasonable approximation
{ x: -dotSize/2, y: -dotSize/2, width: dotSize, height: dotSize }
```

Note: The exact CS2 dot size formula isn't documented in public sources. Using `thickness * 2` is a reasonable visual approximation. [ASSUMED — empirical validation may adjust the multiplier]

### Outline

Rendered as a slightly larger duplicate of each arm/dot underneath (rendered first), black fill, offset by `outline.thickness` px on all sides:

```typescript
// Outlined arm rect expands by outlineThickness in all directions:
{
  x: gap - outlineThickness,
  y: -(thickness/2) - outlineThickness,
  width: armLength + outlineThickness * 2,
  height: thickness + outlineThickness * 2
}
```

### Color

```typescript
const fill = `rgba(${color.r}, ${color.g}, ${color.b}, ${color.a / 255})`
const outlineFill = `rgba(0, 0, 0, ${color.a / 255})`  // black, same alpha as crosshair
```

### Complete SVG structure

```tsx
// Source: UI-SPEC.md CrosshairPreview section + SVG coordinate derivation
<svg
  width={200}
  height={200}
  viewBox="-100 -100 200 200"
  aria-label="Crosshair preview"
>
  {/* Outline layer first (behind) */}
  {outline.enabled && <OutlineArms ... />}
  {/* Arms */}
  <Arms ... />
```

```tsx

interface SliderRowProps {
  label: string;
  min: number;
  max: number;
  step: number;
  value: number;
  onChange: (v: number) => void;
}

function SliderRow({ label, min, max, step, value, onChange }: SliderRowProps) {
  const [inputStr, setInputStr] = useState(String(value));

  // Keep input text in sync when value changes externally
  useEffect(() => { setInputStr(String(value)); }, [value]);

  return (
    <div className="flex items-center gap-2 h-10">
      <span className="shrink-0 min-w-[120px] text-sm text-[#e5e5e5]">{label}</span>
      <input type="range" min={min} max={max} step={step} value={value}
        onChange={(e) => onChange(Number(e.target.value))}
        className="flex-1 ..." />
      <input type="number" value={inputStr} min={min} max={max} step={step}
        onChange={(e) => setInputStr(e.target.value)}
        onBlur={() => {
          const n = parseFloat(inputStr);
          const clamped = isNaN(n) ? value : Math.min(max, Math.max(min, n));
          setInputStr(String(clamped));
          onChange(clamped);
        }}
        onKeyDown={(e) => { if (e.key === 'Enter') e.currentTarget.blur(); }}
        className="w-14 text-right text-sm ..." />
    </div>
  );
}
```

### Anti-Patterns to Avoid

- **Store reads in leaf components:** Only CrosshairDesigner and CrosshairPreview read useCrosshairStore. Other components receive props.
- **Debouncing slider onChange:** The UI-SPEC explicitly states no debounce — all computation is synchronous. Adding debounce breaks the "real-time" requirement.
- **Using `display: none` via conditional rendering:** The UI-SPEC locks `display: none` for hidden controls. Use conditional class application or a wrapper with `hidden` class, NOT unmounting the component entirely (which loses local state). Implement as: `<div className={visible ? '' : 'hidden'}>`.
- **Using `disabled` instead of hidden:** CONTEXT.md is explicit: controls that don't apply to the selected style are **hidden**, not disabled.
- **Native `<input type="color">` in Phase 2:** ColorPicker primitive uses native color picker (hex only, no alpha). Phase 2 color section uses R/G/B/A Slider rows instead — do not use ColorPicker here.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

## Tailwind v4 Form Styling — Range Input Specifics

[VERIFIED: project setup — no tailwind.config.js, CSS-first via `@import "tailwindcss"`]

### The trap: range input pseudo-elements

Range inputs (`<input type="range">`) require vendor-prefixed pseudo-elements that Tailwind v4 utilities do not expose as standalone classes:

- `::-webkit-slider-thumb` — Chrome/Safari thumb
- `::-webkit-slider-runnable-track` — Chrome/Safari track
- `::-moz-range-thumb` — Firefox thumb
- `::-moz-range-track` — Firefox track

### Solution: `@layer` block in `src/index.css`

Since there is no `tailwind.config.js` (Tailwind v4 is CSS-first), add custom styles directly in `src/index.css` after the `@import`:

```css
/* src/index.css */
@import "tailwindcss";

@layer components {
  .slider-range {
    -webkit-appearance: none;
    appearance: none;
    height: 4px;
    border-radius: 2px;
    background: #3a3a3a;
    outline: none;
    cursor: pointer;
  }

  .slider-range::-webkit-slider-thumb {
    -webkit-appearance: none;
    width: 16px;
    height: 16px;
    border-radius: 50%;
    background: #e5e5e5;
    cursor: pointer;
  }

  .slider-range::-moz-range-thumb {
    width: 16px;
    height: 16px;
    border-radius: 50%;
    background: #e5e5e5;
    border: none;
    cursor: pointer;
  }

  .slider-range:focus-visible::-webkit-slider-thumb {
    outline: 2px solid #f97316;
    outline-offset: 2px;
  }
}
```

Apply to Slider component via `className="slider-range w-full"`.

## Common Pitfalls

### Pitfall 1: Slider `step` precision causing float drift

**What goes wrong:** `line.gap` uses `step=0.5`. After many increments, JavaScript floating point can produce values like `0.30000000000000004`.
**Why it happens:** IEEE 754 arithmetic on 0.5-step increments.
**How to avoid:** Round store values to 1 decimal place on setter calls: `Math.round(v * 10) / 10`.
**Warning signs:** Number input shows unexpected trailing digits.

### Pitfall 2: Negative gap in SVG — arms cross center incorrectly

**What goes wrong:** When `line.gap` is negative (e.g., -3), the arm start position `gap` is negative, meaning arms extend past center. If not handled, outlines and dots render behind arms or clip incorrectly.
**Why it happens:** SVG renders in document order — later elements paint over earlier ones.
**How to avoid:** Always render outline rects first, then arm rects, then dot rect. Negative gap is valid and expected (CS2 default is -3).

### Pitfall 3: Style reset wiping user's color while switching

**What goes wrong:** Resetting to `STYLE_DEFAULTS` on style change resets color back to default green even if user set a custom color.
**Why it happens:** CONTEXT.md says style switch resets ALL params to CS2 defaults — this is intentional. Don't "preserve" color across style switches.
**How to avoid:** Call `useCrosshairStore.getState().set(STYLE_DEFAULTS[newStyle])` which sets the full state including color. This is the spec behavior.

### Pitfall 4: `display: none` via conditional render vs hidden class

**What goes wrong:** Conditionally rendering `{showFollowRecoil && <SliderRow ... />}` unmounts the component, losing the `useState` for the companion number input.
**Why it happens:** React unmounts/remounts when conditional render changes.
**How to avoid:** Always render the SliderRow but apply `className={visible ? '' : 'hidden'}` to a wrapper div. Component state (number input's local inputStr) is preserved.

### Pitfall 5: Tailwind v4 peer modifier child scope

**What goes wrong:** `peer-checked:translate-x-4` applied to a grandchild of the peer doesn't work.
**Why it happens:** Tailwind `peer` uses CSS sibling combinator — only direct next siblings or descendants of siblings are affected.
**How to avoid:** Ensure the ToggleSwitch DOM structure has the `input.peer` immediately before the wrapper div. The thumb div inside the wrapper gets the `peer-checked:translate-x-4` via CSS `has-[]` on the wrapper, or restructure to use a sibling relationship.

Alternative that avoids the issue entirely: use a single `button` with `onClick` and manage checked state with a `data-checked` attribute + CSS `[data-checked=true]:` variants.

### Pitfall 6: TypeScript `noUnusedLocals` / `noUnusedParameters` — strict mode

**What goes wrong:** `tsconfig.app.json` has `noUnusedLocals: true` and `noUnusedParameters: true`. Destructuring store values you don't immediately use in a render will cause build errors.
**Why it happens:** Strict TypeScript enforces this at build time.
**How to avoid:** Only destructure store fields you actually use in each component. Prefix intentionally unused params with `_` if needed.

### Pitfall 7: Zustand selector — unnecessary re-renders

**What goes wrong:** `const state = useCrosshairStore()` in CrosshairDesigner subscribes to the entire store; any nested field change causes a full re-render of the designer tree.
**Why it happens:** Default Zustand subscription re-renders on any state change.
**How to avoid:** For this phase, full-store subscription is acceptable since all state is relevant to the designer. CrosshairPreview also needs full state. Do not over-optimize — the computation is synchronous and cheap.

```tsx
export function CrosshairPreview() {

  const armLength = line.length * 4;           // 1 CS2 unit = 4px
  const gap = line.gap;                         // 1:1 to px
  const thickness = Math.max(1, line.thickness); // minimum 1px
  const fill = `rgba(${color.r},${color.g},${color.b},${color.a / 255})`;
  const outlineFill = `rgba(0,0,0,${color.a / 255})`;
  const ot = outline.enabled ? outline.thickness : 0;

  // Arm descriptor: [x, y, width, height]
  type Arm = [number, number, number, number];
  const arms: Arm[] = [
    [gap, -thickness / 2, armLength, thickness],                          // right
    [-(gap + armLength), -thickness / 2, armLength, thickness],           // left
    [-thickness / 2, gap, thickness, armLength],                          // bottom
    // top arm omitted when tStyle
    ...(tStyle ? [] : [[- thickness / 2, -(gap + armLength), thickness, armLength]] as Arm[]),
  ];

  return (
    <div className="w-full min-h-[200px] bg-[#1a1a1a] flex items-center justify-center">
      <svg width={200} height={200} viewBox="-100 -100 200 200" aria-label="Crosshair preview">
        {/* Outline layer */}
        {outline.enabled && arms.map(([x, y, w, h], i) => (
          <rect key={`ol-${i}`}
            x={x - ot} y={y - ot}
            width={w + ot * 2} height={h + ot * 2}
            fill={outlineFill} />
        ))}
        {/* Arms */}
        {arms.map(([x, y, w, h], i) => (
          <rect key={`arm-${i}`} x={x} y={y} width={w} height={h} fill={fill} />
        ))}
        {/* Center dot */}
        {dot.enabled && (() => {
          const ds = thickness * 2;
          return <rect x={-ds / 2} y={-ds / 2} width={ds} height={ds} fill={fill} />;
        })()}
      </svg>
    </div>
  );
}
```

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| XSS via user-controlled SVG fill attribute | Tampering | CSS `rgba()` value is built from store integers (r, g, b, a) — all parsed as `Number()`, never raw strings interpolated into HTML; safe |

## Open Questions

1. **Exact CS2 per-style default values**
   - What we know: CS2 has defaults per style; the store's DEFAULT_CROSSHAIR uses style 4 with gap -3, size 4, thickness 1
   - What's unclear: Whether Classic (style 2) and Tee (style 5) have different CS2 defaults for gap/size
   - Recommendation: Use the same gap/size/thickness defaults for all styles initially; if empirically wrong, update STYLE_DEFAULTS in a follow-up. This does not block the phase.

2. **Slider className passthrough — modify primitive or wrap?**
   - What we know: Slider.tsx has no `className` prop; SliderRow will need a styled range input
   - What's unclear: Whether modifying the Slider primitive conflicts with any future usage
   - Recommendation: Add `className?: string` to Slider's props and pass it to the `<input type="range">`. This is additive and non-breaking.
